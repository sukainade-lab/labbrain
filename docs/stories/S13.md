# S13 — Document versioning: replace a document + re-index without losing Q&A history

> **Status spec (per lesson L8):** the BRD lists S13 only as a backlog one-liner
> (`brd.md:169` — "Document versioning — replace a document and re-index without losing
> Q&A history", loop: domain). This file enumerates the checkable `AC-13.N` acceptance
> criteria that `@AC-13.N` tests trace to. Written **before** `/3-eo-code`.

**Story:** As the Technical/Quality Manager of a JISM-accredited lab, when an SOP or a
calibration procedure is revised, I want to **replace** the document already in LabBrain with
the new revision and have the assistant re-index it automatically — **without losing the
record of questions we already asked**. The library entry stays the same entry (same link,
same place in the list); only its contents and a version number change. A revision that fails
to parse must not destroy the working copy we already had.

**ICP:** the lab admin/owner (Arabic-native, Jordanian dialect, WhatsApp-first, 375px
screen-share). They expect: a one-step "استبدال" (replace) on the document row, a clear
"this is now version 2" signal, the assistant to keep answering from the new revision, and the
Q&A audit trail (S9 export) to remain intact.

## Why this shape (the core insight)

Q&A history is **already decoupled** from documents in the schema, by design:

- `queries.citations` is `jsonb not null default '[]'` — a **frozen snapshot** built at answer
  time by `buildCitations` (`document_id`, `document_name`, `section`, `page_number`,
  `similarity`). See `src/lib/qa/citations.ts`.
- `queries` has **no foreign key** to `documents` (migration 0001). Deleting a document never
  deletes a query; replacing one never rewrites a prior citation.

So "without losing Q&A history" is satisfied **by construction** — there is nothing to migrate.
S13 is therefore (1) a safe *replace + atomic re-index* that keeps the same `document_id`, and
(2) an explicit live test proving history + citations are untouched across a replace.

## Architecture decisions (locked, `/2-eo-dev-plan story-13`)

- **Same `document_id`, monotonic `version`.** A replace mutates the existing row in place: same
  id, `version` incremented, `updated_at` set, `created_at` preserved. The library row, its
  `?doc=ID` citation deep-link, and analytics identity all stay stable.
- **Atomic chunk swap via an RPC.** The Supabase JS client can't run a multi-statement
  transaction, so `replace_document_chunks(document_id, tenant_id, rows jsonb)` (migration 0014,
  `security definer`, pinned `search_path`) does `delete` old + `insert` new in **one** function
  body. Embeddings travel as the existing `toVectorLiteral` text and are cast `::vector` inside.
  This is the only place a document's chunks ever change during a replace — all-or-nothing.
- **Fail-safe replace.** Parse + embed happen first (off the response path); the destructive
  swap runs only after the new chunks are ready. If parse/embed fails, the old file + old chunks
  + old `version` remain (status surfaced as `failed`); the prior revision stays retrievable.
- **Storage replace-in-place.** The new file is stored under `{tenant_id}/{document_id}/{filename}`
  (same namespace, same RLS path policy from migration 0003). If the new filename differs from
  the old, the old object is removed (mirrors S12 logo replace-in-place) so a document never
  accumulates orphaned files.
- **No prior-version blobs kept.** The BRD asks to preserve *Q&A history*, not prior file bytes.
  A version counter + preserved queries satisfies it; keeping every past revision's file is out
  of scope (deferred).
- **Same accepted formats/cap and same tenant-member access as upload.** Replace reuses
  `uploadMetaSchema`/`resolveMime`/`MAX_UPLOAD_BYTES` (PDF/DOCX/XLSX, ≤50 MB) and is gated like
  the existing DELETE (any signed-in member of the owning tenant; cross-tenant → 403).

## Acceptance criteria

### AC-13.1 — Replace a document from the library
- A per-row "استبدال" control on `(app)/documents/page.tsx` uploads a new file via
  `PUT /api/documents/[id]` (multipart). Accepted: PDF, DOCX, XLSX; max **50 MB**.
- The API validates: oversize → **413**; unsupported type / missing file → **400** (Arabic
  message). A valid replace → **200** and the document re-enters the `parsing → indexing →
  ready` pipeline the UI already polls.

### AC-13.2 — Same id; version bump (migration 0014)
- Migration `0014_document_versioning.sql` adds `version int not null default 1` and
  `updated_at timestamptz` to `documents`.
- A successful replace keeps the **same `document_id`**, increments `version`, sets `updated_at`,
  and preserves `created_at`. The `?doc=ID` citation deep-link keeps resolving to the row.

### AC-13.3 — Atomic re-index (chunk swap)
- On replace the new file is parsed + embedded, then the document's old `document_chunks` are
  replaced by the new set via `replace_document_chunks` in **one transaction**.
- After a successful replace, retrieval returns only the new revision's chunks; `page_count`
  reflects the new file. No window where old and new chunks coexist for the document.

### AC-13.4 — Q&A history preserved (headline)
- A replace (and a delete) never modifies or deletes any `queries` row or its `citations`.
- A question asked before a replace still shows its original citation snapshot (document name +
  page captured at answer time). Proven by a live test.

### AC-13.5 — Safe on failure
- If parsing or embedding the new file fails, the document keeps its **previous file, previous
  chunks, and previous `version`**; status is surfaced as `failed`. No partial swap — old chunks
  are deleted only inside the same successful RPC call that inserts the new ones.

### AC-13.6 — Storage hygiene + isolation
- The new file is stored under `{tenant_id}/{document_id}/{filename}`. If the new filename
  differs from the old, the old object is removed (no orphans).
- Storage RLS path isolation (migration 0003) is unchanged: a tenant can only write under its
  own prefix.

### AC-13.7 — Auth-gated, tenant-scoped (L1)
- `PUT /api/documents/[id]` requires a session (**401**), a document that exists (**404**) and
  belongs to the caller's tenant (**403** cross-tenant). The tenant is resolved from the session
  (`users.tenant_id`), never from the client. Each branch (401/404/403/400/413/200) has a
  route-handler integration test (L1).

### AC-13.8 — RTL, 375px, accessible (L7/L9)
- The replace control + version badge render correctly RTL at 375px with ≥44px tap targets,
  `<bdi>` on the (possibly renamed) filename and the `نسخة N` badge (L5).
- `/documents` is added to the axe-walk `DEFAULT_ROUTES`; `npm run axe:walk` reports **0
  serious/critical** there. Any AA failure on the row's own controls is fixed inline before
  scoring (L9).

## Out of scope (defer)
- Keeping prior file revisions / version history browsing / diff (we keep only the current file
  + a version counter; Q&A history is what's preserved).
- Restore/rollback to a previous version.
- Concurrent replace of the same document (single-writer assumption per doc, as with upload).
- Replace-time re-validation of an in-flight upload's parse (reuses the S2 pipeline as-is).
