**Story:** As the Technical/Quality Manager of a JISM-accredited lab, I want my documents organised
into **service panels** — a permanent **خدماتي الحالية** (Existing Services) panel for what I'm
already accredited for, and **خدمة جديدة** (New Service) tabs I add when exploring a new scope —
and to upload each file into the right section, **so that** LabBrain knows which documents belong to
which service and (in S19) can answer scoped to each. The documents already in LabBrain must keep
working — they move into Existing Services automatically, with nothing lost.

**ICP:** the lab admin/owner (Arabic-native, Jordanian dialect, WhatsApp-first, 375px screen-share).
They expect: a clear two-panel layout that reads right-to-left, a one-tap `+` to start a new
service and `✕` to drop one, per-section upload, and every document they already uploaded still
present (now under Existing Services).

## Why this shape (the core insight)

The retrofit is **purely additive at the data layer**, so it does not reopen the audited
tenant-isolation surface and does not touch retrieval:

- `panel_type` / `service_tab_id` are **new columns** on `documents` and `document_chunks`. Existing
  retrieval (`match_document_chunks`, migration 0004) is unchanged — adding columns it doesn't
  select can't change its results. **Panel-scoped search is deliberately deferred to S19**, which
  will change the RPC; S18 ships the data + UI foundation only.
- Existing rows satisfy the new column **defaults** (`panel_type='existing'`, `service_tab_id=NULL`),
  so the "migrate existing docs into Existing Services" requirement (founder decision) is satisfied
  **by construction** — the backfill is a verified no-op safety net, **zero data loss**.
- Chunks carry a **denormalized** copy of their parent document's `panel_type`/`service_tab_id`
  (stamped at ingest) so S19's vector pre-filter stays a single tenant-scoped index scan with no
  join — the same shape as today's tenant filter.

## Architecture decisions (locked, `/2-eo-dev-plan story-18`)

- **`service_tabs` is its own tenant-scoped table** (not a JSON blob on `tenants`): a New Service tab
  is a first-class entity documents/chunks FK to, with named RLS `tenant_isolation_service_tabs`
  mirroring `documents_tenant_isolation` (migration 0002). Deleting a tab `cascade`s its documents'
  rows; chunks cascade from documents (existing FK). The Existing Services panel is **not** a row —
  it is the implicit `panel_type='existing', service_tab_id IS NULL` partition, always present.
- **Two column vocabularies, validated app-side.** `doc_section` is free `text` (not a DB check) so
  the two section sets — Existing: `sops|references|equipment`; New: `references|available_equipment|
  additional_info` — stay flexible; validity is enforced per `panel_type` in
  `src/lib/validation/workspace.ts`. `panel_type` **is** a DB `check ('existing','new_service')`.
- **Ingest stamps the tags from the parent document.** `createDocument`/`processDocument`
  (`src/lib/documents/ingest.ts`) take `panelType`/`serviceTabId`/`docSection`, write them on the
  `documents` row, and copy `panelType`/`serviceTabId` onto every `document_chunks` row. The pure
  `chunkBlocks` splitter is untouched.
- **Tenant resolved from the session, never the client.** `/api/service-tabs` and the extended
  `/api/documents` POST resolve `tenant_id` via `users.tenant_id` from the auth session; a
  client-supplied `service_tab_id` is accepted only after verifying it belongs to that tenant
  (else 400) — Lab A can never attach a document to Lab B's tab.
- **The right-side Q&A panel is unchanged in S18.** The workspace renders the existing Q&A
  component alongside the new left panel; source-filter pills + panel-scoped retrieval are S19.
- **Upload contract preserved.** Same accepted formats/cap (PDF/DOCX/XLSX ≤ 50 MB,
  `uploadMetaSchema`/`resolveMime`/`MAX_UPLOAD_BYTES`), same `parsing → indexing → ready` pipeline
  and poll, same per-row delete (Storage object + chunk cascade), same doc caps (Starter 50 /
  Pro 200) and upgrade message — now exercised per section.

## Acceptance criteria

### AC-2.1 — Two panel types
- The workspace (`(app)/documents/page.tsx`) shows a tab bar: a **permanent** "خدماتي الحالية"
  (Existing Services) tab and zero-or-more **dynamic** "خدمة جديدة" (New Service) tabs.
- Existing Services sections: **SOPs / References / Equipment**. New Service sections:
  **References / Available Equipment / Additional Info**.
- A `+` control adds a New Service tab (names it, e.g. `خدمة جديدة 1`); each New Service tab has a
  `✕` that removes it after a founder-confirm dialog. No limit on the number of New Service tabs.
- The SOP Draft Generator affordance slot exists only on New Service tabs (rendered as a disabled/
  hidden stub in S18; wired in S20) — never on Existing Services.

### AC-2.2 — Upload pipeline (preserved)
- Upload accepts PDF / DOCX / XLSX up to **50 MB**, stored under the tenant namespace
  (`{tenant_id}/{document_id}/{filename}`). Every file — any panel, any section — goes through the
  same parse + index pipeline. Oversize → **413**; unsupported type / missing file → **400**
  (Arabic message).

### AC-2.3 — Status progression (preserved)
- On upload the row shows `uploading → parsing → indexing → ready` via the existing poll; a parse
  failure surfaces `failed`.

### AC-2.4 — Chunk + document metadata (migration 0017)
- Migration `0017_service_tabs_workspace.sql` adds to `documents`: `panel_type text not null
  default 'existing' check (panel_type in ('existing','new_service'))`, `service_tab_id uuid
  references service_tabs(id) on delete cascade`, `doc_section text not null default 'references'`;
  and to `document_chunks`: `panel_type text not null default 'existing'`, `service_tab_id uuid`.
- On ingest, every `document_chunks` row carries the parent document's `panel_type` and
  `service_tab_id`. A New Service document and its chunks carry that tab's id; an Existing Services
  document carries `panel_type='existing', service_tab_id=NULL`.

### AC-2.5 — Tenant isolation at the vector layer (unchanged, regression-guarded)
- All pgvector queries still filter by `tenant_id` before similarity search; the new columns do not
  widen the isolation surface. `service_tabs` has named RLS `tenant_isolation_service_tabs`
  (`for all using/with check (tenant_id = current_tenant_id())`). A live test proves Lab A cannot
  read/insert Lab B's service_tabs, documents, or chunks.

### AC-2.6 — Per-file controls + delete (preserved)
- Each file in a section shows name, upload date, status, and a delete control. Delete removes the
  Storage object and all the document's `document_chunks` (existing FK cascade). Q&A history is
  untouched (S13 contract — `queries` has no FK to `documents`).

### AC-2.7 — Doc caps (preserved)
- The workspace enforces the tenant-wide cap (Starter 50 / Pro 200) across **all** panels/sections
  combined; uploading beyond it shows the Arabic upgrade message (existing `assertDocAvailable`).

### AC-2.8 — Migrate existing documents into Existing Services (retrofit-only, founder decision)
- All documents and chunks present before this migration become Existing Services
  (`panel_type='existing'`, `service_tab_id=NULL`) with **zero data loss**; default section =
  `references`. The migration verifies pre/post row counts are equal. A live test seeds a
  pre-retrofit-shaped doc + chunks and asserts they appear under Existing Services and remain
  retrievable.

### AC-2.9 — Service-tab CRUD API, auth-gated, tenant-scoped (L1)
- `GET /api/service-tabs` lists the caller's tenant's tabs; `POST` creates one (validated name →
  **201**); `DELETE` removes one the caller's tenant owns (cascades its docs/chunks → **200**).
  Branches: no session → **401**; invalid/empty name → **400**; foreign/unknown `service_tab_id` →
  **400/404**. Tenant resolved from session. Each branch has a route-handler test (L1). The
  extended `/api/documents` POST accepts + validates `panel_type`/`service_tab_id`/`doc_section`
  (New Service requires a valid owned `service_tab_id`).

### AC-2.10 — RTL, 375px, accessible (L5/L7/L9)
- The whole workspace is RTL-first, renders correctly at **375px** (tab bar wraps/scrolls; `+`/`✕`
  and upload controls ≥ **44px** tap targets), light theme (BG `#F8FAFC`, navy `#1B2A3D`, amber
  `#D97706`). `<bdi>` wraps every filename **and** every user-supplied service-tab name (mixed
  Latin/Arabic) from first implementation (L5). `npm run axe:walk` @375px reports **0
  serious/critical** on `/documents`; any AA contrast/target failure on the workspace's own
  controls is fixed inline before scoring (L9).
