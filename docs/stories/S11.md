# S11 — Air-gap mode: self-hosted LlamaParse + Ollama (classified documents)

> **Status spec (per lesson L8):** the BRD lists S11 only as a backlog one-liner
> (`brd.md:167` — "Air-gap mode — self-hosted LlamaParse + Ollama for labs with
> classified documents", loops: domain, compliance). This file enumerates the
> checkable `AC-11.N` acceptance criteria that `@AC-11.N` tests trace to. Written
> **before** `/3-eo-code`.

**Story:** As a lab that handles classified / national-security / defence-adjacent
documents (a real segment of JISM-accredited ISO 17025 testing labs), I need to run
LabBrain entirely on my own network — document parsing, embeddings, and answer
generation all performed by local services — so that **no document text or derived
data ever leaves my premises**. A single deploy-time switch flips the whole inference
stack from cloud (OpenAI + cloud LlamaParse) to local (Ollama + self-hosted
LlamaParse), and the system **refuses to operate** rather than silently fall back to
a cloud provider if the local stack is misconfigured.

**ICP:** the on-prem operator / IT-responsible Quality Manager at a security-sensitive
lab. They (or their integrator) deploy LabBrain on a Contabo-style or fully on-prem
host alongside an Ollama instance and a self-hosted LlamaParse, set `INFERENCE_MODE=
airgap`, and need a way to *confirm* the no-cloud guarantee is in force.

## Why now / why this shape

- **It's the named compliance unlock.** The product's core promise is source-traced,
  zero-hallucination answers over a lab's *own* documents (`CLAUDE.md` safety
  contract). For labs whose documents are classified, the residual objection is that
  parsing/embeddings/answers transit cloud processors (OpenAI, cloud LlamaParse —
  "processing-only / transient", but still off-premises). Air-gap mode removes that
  objection entirely.
- **The seams already exist.** `src/lib/parsing/llamaparse.ts` already honours
  `LLAMAPARSE_BASE_URL`; `src/lib/ai/embeddings.ts` and `src/lib/ai/answer.ts` use
  the OpenAI SDK, and **Ollama exposes an OpenAI-compatible API** (`/v1/embeddings`,
  `/v1/chat/completions`). So this is a **backend-resolution + fail-closed-guard**
  story, not a rewrite — and it needs **zero new npm dependencies** (reuse the
  existing `openai` SDK with a `baseURL` override).
- **Same seam discipline as S6/S9/S10.** Just as S6 put each payment rail behind a
  `PaymentProvider` interface and S10 put cross-region transfer behind a
  `MigrationTarget` seam, S11 puts inference-backend selection behind one pure
  resolver (`inference-mode.ts`). CI/tests run against the resolver + mocked HTTP;
  no live Ollama/LlamaParse instance is required to test.

## Architecture decisions (locked, `/2-eo-dev-plan story-11`)

- **One source of truth.** A pure module `src/lib/ai/inference-mode.ts` reads
  `INFERENCE_MODE` (`cloud` | `airgap`, default `cloud`) and resolves, per seam, the
  `{ baseURL, model, apiKey }` to use. Nothing else branches on the mode — the three
  seams ask the resolver.
- **Deploy-time, not per-tenant.** Air-gap is a property of the whole deployment
  (the host either is on a classified network or isn't). No runtime/per-tenant toggle
  — that would invite a mixed cloud/local state, defeating the guarantee.
- **Fail closed, never fall back.** If `INFERENCE_MODE=airgap` and any required local
  var is missing/invalid, the resolver **throws** — the operation errors loudly. It
  must be impossible for an air-gap deployment to silently route a classified document
  to a cloud endpoint. (In `cloud` mode the existing cloud behaviour is unchanged.)
- **Embedding-dimension safety is enforced, not assumed.** The pgvector column is
  `vector(1536)` (baked into migrations). The resolver validates that the configured
  air-gap embedding model's declared dimension (`AIRGAP_EMBEDDING_DIM`) equals
  `EMBEDDING_DIM` (1536) and refuses otherwise — turning a silent vector-corruption /
  retrieval-failure footgun into an actionable startup error.
- **Ollama via the OpenAI-compatible API.** Embeddings and answers reuse the `openai`
  client pointed at `OLLAMA_BASE_URL` with a dummy key (Ollama ignores auth). No new
  dependency, no second client abstraction.
- **No DB migration.** This story changes inference backends only — no schema change.

## Acceptance criteria

### AC-11.1 — Single mode switch resolves all three seams
- `src/lib/ai/inference-mode.ts` exports a resolver reading `INFERENCE_MODE`
  (`cloud` | `airgap`; unset/empty → `cloud`; any other value → throws an actionable
  error — no silent default to cloud on a typo'd `airgap`).
- It returns the backend config for each of the three seams (parse / embed / answer):
  `{ baseURL, model, apiKey }`. The three seams (`llamaparse.ts`, `embeddings.ts`,
  `answer.ts`) obtain their backend **only** from this resolver — no mode branching
  scattered elsewhere.

### AC-11.2 — Air-gap embeddings go to local Ollama only
- In `airgap` mode, `embedTexts()` calls the Ollama OpenAI-compatible endpoint
  (`OLLAMA_BASE_URL` + `AIRGAP_EMBEDDING_MODEL`); the request host is the configured
  local host and is **never** `api.openai.com`.
- The returned vectors preserve order and dimension exactly as the cloud path does
  (the `toVectorLiteral` contract is unchanged).

### AC-11.3 — Air-gap answers go to local Ollama only; grounding contract unchanged
- In `airgap` mode, `generateAnswer()` calls Ollama chat (`AIRGAP_ANSWER_MODEL`);
  the request never reaches OpenAI/Claude cloud.
- The grounding contract is byte-for-byte unchanged from the cloud path: same system
  prompt (`buildSystemPrompt`), `temperature: 0`, same context formatting, and the
  no-chunk path (AC-3.5) still never reaches the model.

### AC-11.4 — Air-gap parsing targets the self-hosted LlamaParse
- In `airgap` mode, `parseDocument()` resolves its base URL from the resolver
  (self-hosted `LLAMAPARSE_BASE_URL`); the cloud default base
  (`https://api.cloud.llamaindex.ai/...`) is **never** used.
- The page→block citation contract (one block per page, heading as section) is
  unchanged.

### AC-11.5 — Fail-closed guard (the core compliance guarantee)
- A pure validation helper rejects an air-gap deployment that is missing any required
  local var (`OLLAMA_BASE_URL`, `AIRGAP_EMBEDDING_MODEL`, `AIRGAP_EMBEDDING_DIM`,
  `AIRGAP_ANSWER_MODEL`, self-hosted `LLAMAPARSE_BASE_URL`) with a clear,
  field-specific error.
- On any such misconfiguration the seams **throw** rather than execute — it is
  impossible to reach a cloud endpoint while `INFERENCE_MODE=airgap`. (Tested:
  air-gap + missing var → throws; never issues a cloud-host request.)

### AC-11.6 — Embedding-dimension match enforced
- The resolver/validator confirms `Number(AIRGAP_EMBEDDING_DIM) === EMBEDDING_DIM`
  (1536). A mismatch throws an actionable error naming both the configured dim and the
  required column dim **before** any embedding is requested — preventing corrupt
  vectors and silent retrieval failure.

### AC-11.7 — Env contract + deploy documentation (no real secret prefixes — L6)
- `.env.example` lists every air-gap var by name only (`INFERENCE_MODE`,
  `OLLAMA_BASE_URL`, `AIRGAP_EMBEDDING_MODEL`, `AIRGAP_EMBEDDING_DIM`,
  `AIRGAP_ANSWER_MODEL`; `LLAMAPARSE_BASE_URL` already present) with abstract
  placeholders — no real-looking secret prefix.
- `docs/env-contract.md` (and a deploy note) documents which service provides each,
  that air-gap is a per-deployment mode, that the embedding model must be 1536-dim,
  and how to run Ollama + self-hosted LlamaParse alongside the on-prem stack.

### AC-11.8 — Operator-visible mode indicator (reachable surface — L4/L5/L7)
- An admin/founder-visible read-only indicator reports the live inference mode:
  "Air-gap — all inference local" (with the resolved embed/answer model names) vs
  "Cloud". It is fed by the resolver, so it cannot disagree with actual behaviour.
- Dynamic model/host strings are wrapped in `<bdi>` for BiDi safety in RTL (L5); the
  surface renders correctly at 375px with ≥44px tap targets and 0 serious/critical
  axe violations (L7).

## Out of scope (defer to future gated stories)
- Live run against a real Ollama / self-hosted LlamaParse instance (a founder/
  integrator-gated deploy step, like the Contabo + me-central-1 cutovers). This story
  ships the *mode + guard + adapters + indicator*, fully tested against mocks.
- Per-deployment **configurable vector dimension** (changing the `vector(1536)`
  column) — a separate migration story; S11 enforces the 1536 match instead.
- Per-tenant air-gap, mixed cloud/local routing, model auto-discovery from Ollama,
  and a local Claude/answer-quality fallback chain.

## Test plan (TDD)
- **Pure (the core):** `inference-mode.ts` resolver + validator —
  `cloud` default (unset/empty), `airgap` happy path, unknown mode → throws,
  per-seam config shape, fail-closed on each missing var (AC-11.5), dim mismatch →
  throws (AC-11.6). No I/O.
- **Seam (mocked HTTP, L1-style):** with `INFERENCE_MODE=airgap`, inject a fake
  `fetch`/client and assert `embedTexts` / `generateAnswer` / `parseDocument` hit the
  **local** host and never an `openai.com` / cloud-LlamaParse host; assert order/dim
  preservation and the unchanged grounding contract; assert air-gap + missing var
  throws before any request is issued.
- **Route (L1):** exercise the air-gap path through the document + query routes with
  the local backend mocked — confirm a misconfigured air-gap deployment surfaces a
  clear error (not a cloud call), and the indicator endpoint/state reflects the mode.
- **L7:** manual 375px + axe-core walk on the mode indicator surface
  (`npm run axe:walk` — the harness landed in #20), RTL + ≥44px targets, 0
  serious/critical violations.
