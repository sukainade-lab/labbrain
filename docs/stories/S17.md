# S17 — Bilingual (cross-lingual) query expansion for Q&A retrieval

> **Status spec (per lesson L8):** the BRD lists S17 as a backlog one-liner (added to
> `brd.md` v2 table — "Bilingual (cross-lingual) query expansion", loop: domain). This file
> enumerates the checkable `AC-17.N` acceptance criteria that `@AC-17.N` tests trace to.
> Written **before** `/3-eo-code`.

**Story:** As the Technical/Quality Manager of a JISM-accredited lab, when I ask a question in
**Arabic** about a standard whose text is in **English** (the common case — ISO/IEC 17025 is
distributed in English), I want LabBrain to find the relevant passage and answer me **in
Arabic, with citations** — not refuse. Today the English version of the same question succeeds
while the Arabic one is refused, even though the answer is in the lab's own document.

**ICP:** Arabic-native lab engineer who reads English standards but thinks and asks in Jordanian
dialect. WhatsApp-first, 375px screen-share. They expect to ask `ما هي الحيادية؟` and get the
ISO 17025 §4.1 ("Impartiality") clause back in Arabic with `📄 [Document] — الصفحة N`.

## Why this shape (the measured gap)

The PR #35 retrieval-gate recalibration (`SIMILARITY_THRESHOLD` 0.75 → 0.35) was validated
**live** in production (`elbsrrtbstxtudnbpvzw`, 2026-06-09): previously-refused questions flipped
to `found_answer = true`. But the measurement surfaced a **distinct** gap:

| Question | Lang | Result vs the same English corpus |
|----------|------|-----------------------------------|
| `what is impartiality?` | en | ✅ found, 2 citations |
| `ما هي الحيادية؟` | ar | ❌ not found |

The content exists (6 chunks mention impartiality; ISO 17025 §4.1 is titled "Impartiality").
Root cause is **cross-lingual retrieval**: an Arabic question vs an English passage for an
abstract term with no shared token scores **below 0.35** cosine with `text-embedding-3-small`.
This is **not** a threshold problem — lowering the gate again would re-admit irrelevant chunks
and is explicitly the mistake the PR #35 fix warned against. The right fix is to **retrieve
with the question in both languages and union the results**.

## Architecture decisions (locked, `/2-eo-dev-plan story-17`, founder-approved)

- **App-side union, ZERO production DDL.** We call the existing, tenant-isolation-tested
  `match_document_chunks` RPC (migration 0004) **twice in parallel** — once with the original
  question's embedding, once with the translated question's embedding — and merge the two result
  sets in TypeScript. The RPC, its `current_tenant_id()` filter, the `security definer` grant,
  and the vector column are all untouched. (Rejected alternative: a new SQL RPC taking two
  embeddings — more powerful but reopens the audited tenant-isolation surface for no benefit.)
- **Translate via the existing answer LLM seam.** `translate.ts` calls
  `resolveOpenAiBackend("answer")` at temperature 0 with a strict instruction to **preserve ISO
  clause numbers, measurement units, accreditation-body names (JISM), and English technical
  terms verbatim** (same invariant the answer system prompt already enforces). AR question →
  English query; EN question → Arabic query. This rides the S11 air-gap seam unchanged (in
  air-gap mode the local Ollama answer model does the translation; nothing reaches the cloud).
- **One batched embed call.** The original + translated query are embedded together via the
  existing `embedTexts([...])` (batched, order-preserving) — one network round-trip, not two.
- **Fail-open to today's behaviour.** If translation throws, returns empty, or returns a string
  equal (case/whitespace-insensitive) to the original, we **skip the second retrieval** and
  behave exactly as the single-embedding path does today. Bilingual expansion can only *add*
  recall; it can never break the existing path or the zero-hallucination contract.
- **Env kill-switch, default ON.** `QA_BILINGUAL_EXPANSION` (default on) gates the whole feature.
  Set it to `0`/`false` to instantly revert to single-embedding retrieval without a deploy. Names
  only in `.env.example` (L6) — it is not a secret.
- **Answer path + audit log unchanged.** The merged chunks feed the **existing** `generateAnswer`
  + `isNotFoundAnswer` + `buildCitations` path. The answer is still in the **original** question's
  language. The `queries` audit row still records the **original** `question_text`/`question_lang`
  only — the translated query is an internal retrieval aid, never persisted or shown to the user.

## Acceptance criteria

### AC-17.1 — Retrieve with both the original and translated query
- When `QA_BILINGUAL_EXPANSION` is on and a non-trivial translation is produced, `ask()` retrieves
  using **both** the original-language question embedding and the cross-language (AR↔EN) translated
  question embedding, and combines the results before answering.

### AC-17.2 — Faithful cross-lingual translation (LLM seam)
- `translate.ts` translates the question to the other language (ar→en, en→ar) via
  `resolveOpenAiBackend("answer")` at temperature 0.
- The translation **preserves verbatim**: ISO clause numbers (e.g. `17025`, `§4.1`), measurement
  units, accreditation-body names (JISM), and embedded English technical terms. It returns the
  bare translated query (no preamble/quotes).

### AC-17.3 — Union + dedupe + top-N (pure `merge.ts`)
- Both query embeddings are produced in **one** batched `embedTexts` call; the two
  `match_document_chunks` RPC calls run in **parallel** (`Promise.all`), each tenant-filtered by
  the RPC exactly as today.
- The two `RetrievedChunk[]` result sets are unioned and **deduped by `id`**, keeping the **max**
  `similarity` per chunk, sorted by `similarity` descending, truncated to `MATCH_COUNT` (5).
  `merge.ts` is a pure function (no I/O) with full unit coverage (empty/empty, overlap keeps max,
  disjoint union capped, stable order).

### AC-17.4 — Answer path + zero-hallucination contract unchanged
- The merged chunks feed the existing `generateAnswer` → `isNotFoundAnswer` canonicalisation →
  `buildCitations` path. The answer is in the **original** question's language. If the model still
  refuses (off-topic excerpts), the exact `NOT_FOUND[lang]` sentinel is returned. No new path can
  produce an ungrounded answer.

### AC-17.5 — Fail-open degradation
- If translation throws, returns empty/whitespace, or equals the original query
  (case/whitespace-insensitive), `ask()` degrades to **single-embedding** retrieval — identical to
  today's behaviour. A translation failure never throws out of `ask()` and never changes the
  user-visible result vs the pre-S17 single-query path.

### AC-17.6 — Audit log records only the original question
- The `queries` row records the original `question_text` and detected `question_lang` exactly as
  before. The translated query is **never** persisted to `queries`, returned in `QaResult`, or
  shown in the UI.

### AC-17.7 — Env-gated, default on (L6)
- `QA_BILINGUAL_EXPANSION` defaults to **on** when unset; `0`/`false`/`off` (case-insensitive)
  disables it and restores single-embedding retrieval. Added to `.env.example` **names-only**
  (no secret prefix). When off, `ask()` issues exactly one embed + one RPC call.

### AC-17.8 — Live proof (the measured gap closes)
- A live test (serialized, unique-tenant, L2) seeds an English corpus containing impartiality
  content, asks `ما هي الحيادية؟` (Arabic) with expansion **on**, and asserts the result is
  `found = true` with **≥1 citation** — and that with expansion **off** against the same corpus it
  is refused (demonstrating the expansion, not the threshold, is what closes the gap).

## Out of scope (defer)
- Translating into more than the two project languages (ar/en) — the ICP + corpus are AR/EN only.
- Caching translations (each question is embedded fresh; volume is low; revisit if cost matters).
- A UI surface for the feature — it is invisible by design (better recall, same UX). No new UI.
- Re-ranking beyond max-similarity union (e.g. RRF) — union+max is sufficient for the measured gap.

## DB / migrations
- **None.** S17 introduces **zero** schema or RPC changes — it composes the existing
  `match_document_chunks` RPC (migration 0004). This is the central architecture choice.
