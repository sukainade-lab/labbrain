# S12 — Multi-tenant branding: lab logo in the UI header

> **Status spec (per lesson L8):** the BRD lists S12 only as a backlog one-liner
> (`brd.md:168` — "Multi-tenant branding — lab logo in UI header", loop: domain).
> This file enumerates the checkable `AC-12.N` acceptance criteria that `@AC-12.N`
> tests trace to. Written **before** `/3-eo-code`.

**Story:** As the Technical/Quality Manager of a JISM-accredited lab, I want my own
lab's logo (and name) shown in the LabBrain app header instead of the generic
"LabBrain" wordmark, so the tool feels like *our* system when I screen-share it in an
accreditation review or onboard a colleague. An admin uploads the logo once from
Settings; every member of the lab then sees it across the authenticated app.

**ICP:** the lab admin/owner (Arabic-native, Jordanian dialect, WhatsApp-first,
375px screen-share demos). They expect a one-step upload, an obvious "this is your
logo" preview, and a way to remove/replace it. Non-admin members see the branding but
cannot change it.

## Why now / why this shape

- **First net-new visible-UI story since the axe-walk harness (#20) landed.** L5/L7/L9
  all apply to fresh surface — the chance to ship a polished, accessible RTL feature
  first-pass and earn above the mock-tested-infra 90 plateau.
- **The patterns already exist.** Tenant-namespaced Storage + path-isolating RLS is
  exactly migration 0003's `documents` bucket pattern; multipart upload + validate +
  admin-client store is the `/api/documents` pattern; the Settings section mirrors
  `<MfaSettings>`. So this is **composition of proven seams**, not new infrastructure.
- **Multi-tenant safety still governs.** A lab must never be able to read or overwrite
  another lab's logo — the same tenant-isolation contract as documents/chunks. Storage
  RLS keys on the first path segment = `current_tenant_id()`.

## Architecture decisions (locked, `/2-eo-dev-plan story-12`)

- **One logo per tenant, replace-in-place.** `tenants.logo_path` (nullable) holds the
  single current object key. Upload removes any prior object first, so a tenant never
  accumulates orphans and the header has exactly one source.
- **Dedicated public `branding` bucket.** Logos are not sensitive and are rendered in a
  server-rendered header on every app page; a public-read bucket gives a stable URL
  with no per-request signed-URL refresh. **Writes stay RLS-gated** (tenant-path
  policy), so public-read does not weaken tenant isolation.
- **Admin-only mutation.** Upload/remove require the caller's `users.role` ∈
  (`owner`, `admin`). Members get 403. The tenant is always resolved from the session
  (`users.tenant_id`), never trusted from the client (AC-12.8).
- **Validation mirrors documents.** A pure `validation/branding.ts` caps size at
  512 KB and restricts MIME to PNG/JPEG/SVG/WebP — small, image-only, unit-tested.
- **Served via `<img src>` only.** The logo (including SVG) is referenced by URL on the
  Supabase storage origin and never inlined into the DOM, so an uploaded SVG cannot
  execute script in the app origin.

## Acceptance criteria

### AC-12.1 — Admin uploads a logo from Settings
- A Branding section in `/settings` lets an admin upload an image. Accepted: PNG, JPEG,
  SVG, WebP; max **512 KB**.
- The API validates: oversize → **413**; unsupported type / missing file / bad shape →
  **400**; each with an Arabic message. A valid upload → **201** with the new logo URL.

### AC-12.2 — Tenant-namespaced storage with isolating RLS
- The logo is stored in a `branding` bucket under `{tenant_id}/logo.{ext}`.
- A storage RLS policy (`branding_storage_tenant_isolation`) restricts authenticated
  write/all to objects whose first path segment equals `current_tenant_id()`. Lab A can
  never write or overwrite Lab B's logo path. (Tested live, serialized — L2.)

### AC-12.3 — `tenants.logo_path` column (migration 0013)
- Migration `0013_tenant_branding.sql` adds `logo_path text` (nullable) to `tenants`
  and creates the public `branding` bucket + the RLS policy above.
- A successful upload sets `tenants.logo_path` to the new key, replacing (and deleting)
  any prior object so at most one logo exists per tenant.

### AC-12.4 — Header renders the tenant logo + name (reachable surface — L4/L5)
- The authenticated app shell (`src/app/(app)/layout.tsx`) renders the tenant's logo
  and lab name in the sidebar, replacing the static "LabBrain" wordmark.
- Fallback chain: logo present → `<img>` + name; no logo → lab name only; no name →
  the "LabBrain" wordmark. The lab name is wrapped in `<bdi>` (L5) so a Latin lab name
  never reorders inside the RTL sidebar.

### AC-12.5 — Admin can remove the logo
- A remove action deletes the bucket object and sets `tenants.logo_path` to null; the
  header falls back to the lab-name/wordmark. DELETE → **200**.

### AC-12.6 — Stable public URL + safe, non-shifting image
- The logo is served from the public `branding` bucket via a stable public URL.
- The header `<img>` carries a meaningful Arabic `alt` (e.g. `شعار <bdi>{lab}</bdi>`)
  and explicit width/height so it cannot cause layout shift.

### AC-12.7 — RTL, 375px, accessible (L7/L9)
- The Branding section and the header logo render correctly RTL at 375px with ≥44px tap
  targets, and pass axe-core with **0 serious/critical** violations (cited via
  `npm run axe:walk`). Any AA failure on the section's own controls is fixed inline
  before scoring (L9).

### AC-12.8 — Auth-gated, admin-only, tenant from session (L1)
- `POST` and `DELETE /api/branding` require a session (401 if absent) and an admin role
  (403 for members). The tenant is resolved from `users.tenant_id` for the signed-in
  user — never from a client-supplied value. Each branch (201/400/413/403/401 on POST;
  200/403/401 on DELETE) has a route-handler integration test (L1).

## Out of scope (defer)
- Per-user / per-document branding, custom colour themes, custom fonts, white-label
  email templates, custom domains (S12 is the header logo + name only).
- Image processing (resize/crop/optimize) — we store the uploaded bytes as-is within
  the size cap; a future story can add server-side normalization.
- Branding on the public marketing/auth pages (those keep the LabBrain wordmark — the
  logo is a *tenant* asset shown only inside the authenticated app).

## Test plan (TDD)
- **Pure:** `validation/branding.ts` — accepted MIME set, 512 KB cap (boundary), bad
  type → error, empty → error, `extForMime`/`resolveMime` round-trip (AC-12.1).
- **Service (mocked admin client):** `uploadLogo` removes prior object → uploads
  `{tenant}/logo.{ext}` (upsert) → sets `logo_path`; `removeLogo` deletes + nulls;
  `publicLogoUrl` builds the URL (AC-12.3/12.5/12.6).
- **Route (L1):** `POST /api/branding` — 401 no session, 403 member, 400 bad type,
  413 oversize, 201 happy; `DELETE` — 401, 403, 200. Admin client + auth mocked
  (AC-12.1/12.5/12.8).
- **Live (L2, serialized, unique-tenant):** branding storage RLS forbids a tenant from
  writing/reading another tenant's `{other_tenant}/logo.*` path (AC-12.2).
- **L7:** `npm run axe:walk` @375px now also exercises `/settings` (Branding section) +
  the header logo — 0 serious/critical.
