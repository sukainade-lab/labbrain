# S10 — KSA data migration: per-tenant move to AWS me-central-1 (PDPL)

> **Status spec (per lesson L8):** the BRD lists S10 only as a backlog one-liner
> (`brd.md:166`). This file enumerates the checkable `AC-10.N` acceptance criteria
> that `@AC-10.N` tests trace to. Written **before** `/3-eo-code`.

**Story:** As the platform founder, when a KSA-based lab needs its data resident in
Saudi Arabia (PDPL — Personal Data Protection Law requires KSA-resident personal
data for some categories), I can migrate exactly one tenant's complete data set from
the EU instance (Supabase Frankfurt) to a KSA-resident instance (AWS me-central-1 /
Riyadh) — with verified parity before any cutover, and an auditable residency record
of who moved what, when, and verified how.

**ICP:** the founder/platform-admin operating on behalf of a KSA paying lab. The
migrated lab's owner/admins are the downstream beneficiaries (their data becomes
PDPL-resident); they do not run the migration themselves.

## Why now / why this shape

- The data-residency map (`architecture/tech-stack-decision.md:88`) already flags:
  *"At KSA launch, migrate to AWS me-central-1 (Riyadh)… Plan for this before the
  first KSA paying lab."* S10 builds that tool.
- **Hard reality of this environment:** we cannot provision a live AWS me-central-1
  Supabase project here. So — exactly as S9 put headless Chromium behind a
  `renderPdfFromHtml()` seam and S6 put each payment rail behind a `PaymentProvider`
  interface — the actual cross-region transfer lives behind a **`MigrationTarget`
  seam**. CI/tests run against an in-memory fake target; the real me-central-1
  connection is a **founder-gated runbook step** (sibling to the Contabo cutover).
  The story ships the *tool + verification + residency record + cutover state
  machine*, fully tested — not a live cross-region run.
- **Reuses S8 founder infrastructure:** access gate (`src/lib/founder/guard.ts`),
  route helpers, and the founder panel as the operable entry point (L4 reachable).

## Architecture decisions (locked, `/2-eo-dev-plan story-10`)

- **`MigrationTarget` seam** — an interface (`upsertRows(table, rows)`,
  `uploadObject(path, bytes)`, `countRows(table, tenantId)`, `checksum(tenantId)`).
  Default impl talks to the KSA Supabase via **service-role** using
  `KSA_SUPABASE_URL` / `KSA_SUPABASE_SERVICE_KEY` (new env, never committed — L6).
  Tests inject an in-memory fake. CI never needs a live KSA project.
- **Single-tenant per run** — least blast radius; migrate one `tenant_id` at a time.
- **Verify-before-cutover, never destructive** — the source stays authoritative
  until parity is proven AND the founder confirms cutover. Source-data *deletion* is
  explicitly **out of scope** (a separate, future founder-gated decommission step).
- **Residency pointer** — `tenants.data_region` (`eu-frankfurt` default →
  `ksa-me-central-1`) is the live routing pointer; the `tenant_migrations` run-log
  row at `status='cutover'` is the PDPL evidence artifact.
- **Deterministic export + content hash** — export rows in stable order and compute
  a checksum over the bundle so source↔target parity is a single comparable value.

## Acceptance criteria

### AC-10.1 — Access control: platform-admin only (fail-closed)
- The migration trigger route (`POST /api/founder/tenants/[tenantId]/migrate`) and
  the status route require an authenticated session **and** an email on the
  `PLATFORM_ADMIN_EMAILS` allowlist — reusing the S8 founder gate.
- Unauthenticated → **401**. Authenticated non-admin → **404** (fail-closed, no
  existence leak — identical posture to the S8 founder panel).
- A malformed `tenantId` (bad uuid) → **400**.

### AC-10.2 — Complete, residency-safe, deterministic export bundle
- The exporter produces a bundle covering **every tenant-scoped table** in FK
  dependency order: `tenants`, `users`, `documents`, `document_chunks` (incl. the
  `embedding` vector), `queries`, `subscriptions`, `invitations`, `audit_exports`,
  plus a **storage-object manifest** (object path + byte size + per-object checksum).
- Every row is scoped to the one `tenant_id` (no other tenant's rows can appear —
  sibling to AC-1.3/AC-2.4; tested explicitly).
- Export runs **in-house / in-region** (server-side, service-role) — no third-party
  service ever sees lab data (PDPL: data does not transit an external processor).
- Rows are emitted in a deterministic order; a **content hash** is computed over the
  serialized bundle for later verification.

### AC-10.3 — Idempotent import into the target (FK + vector fidelity)
- The `MigrationTarget` writes the bundle in FK dependency order, **preserving
  primary keys** so foreign-key relationships survive the move.
- `document_chunks.embedding` vectors round-trip **exactly** (dimension count and
  values preserved — no truncation/precision loss).
- Import is **idempotent**: upsert by primary key, so a retried or partially-failed
  migration converges on re-run and never creates duplicates.

### AC-10.4 — Parity verification before cutover (refuse on mismatch)
- After import, verify **per-table row counts** AND a **content checksum** match
  between source and target for the tenant.
- **Match** → the run becomes eligible for cutover.
- **Mismatch** → the run is marked `failed`, cutover is **refused** (route returns a
  4xx/422 with the diff), and the source remains authoritative — **zero data loss**.

### AC-10.5 — Cutover + residency record (PDPL evidence)
- Cutover (a distinct confirmed action, not implied by a successful import) flips
  `tenants.data_region` to `ksa-me-central-1` **only** when the run reached `verified`.
- The `tenant_migrations` row at `status='cutover'` carries the PDPL evidence:
  `tenant_id`, `source_region`, `target_region`, `verification_hash`, `started_by`
  (admin email), `started_at`, `finished_at`, per-table `row_counts`. This is the
  auditable "who moved what, when, verified how" record.
- **Rollback-safe:** before cutover is confirmed the pointer can revert to the
  source region (source still authoritative). Source data is *frozen/read-only* at
  cutover, never deleted by this story.

### AC-10.6 — Migration run-log: monotonic state machine + isolation
- `tenant_migrations` records each run with `status` ∈
  `pending → exported → imported → verified → cutover` (plus terminal `failed`).
- Status transitions are **monotonic** — a re-invoked run resumes or refuses safely;
  a tenant already at `cutover` cannot be double-migrated (route → 409).
- The migration log is **tenant-isolated** under RLS: a tenant's migration record
  never leaks across tenants (sibling to AC-9.6).

## Out of scope (defer to future gated stories)
- Live cross-region run against a real me-central-1 project (founder-gated runbook,
  like the Contabo cutover).
- Source-data **deletion / decommission** after cutover (separate gated step).
- Bulk/multi-tenant migration, scheduled migration, automatic region selection,
  bidirectional sync, and storage-object byte transfer beyond manifest+checksum
  verification (the seam exposes `uploadObject`; the real byte copy is runbook-gated).

## Test plan (TDD)
- **Pure:** export bundle builder (table coverage, FK order, tenant-scoping,
  deterministic order, content hash stability, embedding round-trip shape); checksum/
  parity comparator (match, count mismatch, checksum mismatch → diff); state-machine
  reducer (legal transitions, monotonic guard, double-cutover refusal).
- **Route (L1):** `POST …/migrate` and status route — 401, 404 non-admin, 400 bad
  uuid, 200 happy path (export→import→verify against the **fake `MigrationTarget`**),
  422 verify-mismatch (cutover refused), 409 already-cutover.
- **Live DB (L2):** serialized, unique-tenant; seed two tenants, assert the exporter
  pulls **only** the target tenant's rows across every table + cross-tenant exclusion;
  assert `tenant_migrations` RLS isolation; `afterAll` cleanup.
- **L7:** manual 375px + axe-core walk on the founder-panel migration control
  (RTL, ≥44px tap targets, 0 serious/critical violations).
